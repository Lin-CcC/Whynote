# TESTING

本文档只记录当前真实可执行的验证命令和手工回归路径，不再保留“仓库尚无业务代码”的早期占位描述。

## 命令基线

- `pnpm run typecheck`
- `pnpm test`
- `pnpm run build`
- `pnpm run test:e2e`（已安装 Playwright / Chromium 时）

## 当前手工回归重点

### 1. 工作区启动与持久化

- 首次打开应用，应能自动初始化最小工作区并进入可编辑状态。
- 修改模块或节点内容后，自动保存状态应能从 `保存中` 回到 `已保存`。
- 刷新页面后，最近一次工作区和当前选中位置应能恢复。

### 2. 文本主视图的选中态与编辑态

- 点击节点卡片：节点应进入选中态，但输入框不应自动抢焦点。
- 点击或聚焦标题 / 内容输入框：节点应继续保持选中，并明确进入编辑态。
- 切换到其他节点后，上一节点的编辑态提示应消失，不残留错误焦点。
- 输入嵌套节点时，只有当前节点自己可以显示“编辑中”；祖先模块 / plan-step 卡片不能因为 capture focus 一起进入编辑态。
- 如果要手工复现这轮抖动问题，优先用“聚焦深层 question 或 answer 的标题 / 正文，再连续输入并观察滚动位置与卡片 chrome”这条路径：
  修前表现：祖先卡片会一起进入编辑态，长标题还会把侧栏上下文卡片撑高，容易出现窗口回跳或焦点附近的布局抖动。
  修后验证：连续输入时焦点不丢、当前节点不乱切；只有当前节点显示编辑态；即使自动保存状态切换，编辑器输入框也不会被重挂载。

### 3. 学习运行时主路径

- 选中 `question` / `answer` 时，主动作应继续落在“评估当前回答”这条闭环路径上。
- 选中一个还没有 `answer` 的 `question` 时，`直接回答当前问题` 应成为唯一明显主动作，`插入回答` 仍保留但降级为手动入口。
- 对手动插入的 `question` 触发 `直接回答当前问题` 后，应生成普通 `answer` 节点，并自动选中这条新回答。
- 对系统生成的 `question` 触发 `直接回答当前问题` 后，也应落成普通 `answer` 并自动选中，不能因为问题来源不同而走不同分支。
- direct answer 完成后，应能继续走 `重新评估当前回答`、`给我提示`、`查看答案解析`、`回到当前回答继续修改` 这条既有 answer-revision 主链。
- 选中 `judgment` 时，正文卡片里的“回到当前回答继续修改”应是唯一明显主动作；提示和答案解析保留为次级动作。
- 选中 `summary / 答案解析` 时，应仍能回到对应回答继续修改，而不是停在说明文本。
- `查看当前步骤完成依据` 的结果应只进入运行状态区，不再额外展开一大片教学说明。

### 4. 运行状态区与 AI 配置

- 运行状态区只需要回答两件事：系统现在忙不忙、最近一条动作结果是什么。
- AI 运行期间，正文编辑和树操作应被临时锁定，并给出简短说明。
- AI 配置中选择模板后，`Base URL / Model` 应自动填充；未填写预设名称时，“保存为新预设”应保持禁用并说明原因。
- 保存本地预设、切换预设、刷新恢复后，三项配置应与本地预设保持一致。
- 删除普通预设时，确认文案应明确说明不会影响当前正在运行的配置值；删除后当前字段值保持不变。
- 删除当前选中预设时，确认后选择器应回到 `当前配置（未保存）`，但 `Base URL / API Key / Model` 保持删除前的值。
- 重命名预设后，`已保存预设` 列表和当前配置选择器应立即显示新名字，且 `Base URL / API Key / Model` 不变。
- 手动修改 `Base URL / API Key / Model` 后，如果之前正在使用某个预设，当前配置应立即脱离该预设并显示为未保存状态。
- 切换厂商模板后，如果之前正在使用某个预设，当前配置也应脱离该预设并显示为未保存状态。

### 5. 资料主路径

- 单份资料录入：支持手动填写，URL 自动补全失败时不阻塞创建。
- 本地资料导入：单文件预填和 `.txt / .md` 批量导入都应可运行；批量导入先预览，再正式导入。
- URL 正文可读时，应优先走“正文基础 + AI 摘要”主路径；不可读时明确回退，不伪装成功。
- `resource / resource-fragment` 的引用、回跳、删除确认和刷新恢复应保持可用。

### 6. 编辑流顺手性回归

- 在任意 `question` 下插入一个空 `answer` 或 `summary`，确认 inspector 里出现轻量“节点类型”切换区；切到另一种合法类型后，标题、正文、引用和标签应保留；如果节点已有子树，或目标类型不符合父节点约束，则对应切换项应禁用。
- 在一个 `question` 下依次准备 `answer`、follow-up `question`、`judgment`、`summary` 等子节点后，再执行手动 `插入总结`；新 summary 应落到这条 question 闭环块尾部，而不是插进前面的 answer/judgment 中间。
- 手写一段 `summary` 后，选中它应出现 `检查这个总结`；执行后生成的应是针对总结理解质量的 `judgment`，文案要覆盖“说对了什么 / 还缺什么 / 可能误解 / 下一步建议”，且不自动生成 follow-up question，也不替代普通 `回答评估` 路径。
- 输入长标题或长正文时，观察两侧栏与状态区：允许文本同步更新，但不应把当前输入焦点打断，也不应因为侧栏文本撑高而制造明显窗口回跳或测试噪音。

### 7. 当前回答语义、过期判定与级联清理

- 先看主视图卡片头部与 `question` 提示行，不要只靠左侧 runtime action card 猜：
  - 当前 answer 卡片会显示 `当前回答`
  - 旧 answer 卡片会显示 `旧回答`
  - question 卡片会显示 `当前回答：...`
- answer-closure judgment / 答案解析 / summary-check judgment 现在都要直接看卡片头部：
  - 最新且仍对应当前主路径的结果显示 `当前结果`
  - 旧轮次或旧回答对应的结果显示 `历史结果`
  - 源 answer / 手写 summary 已改但结果仍保留时，额外显示 `已过期`
  - 配对关系通过 `配对回答：...` 或 `检查对象：...` 直接确认
- 在同一个 `question` 下连续用 `插入回答` 或 `直接回答当前问题` 生成多条 answer 后，runtime 主动作应始终围绕最新晋升的 `currentAnswerId` 工作；即使回头选中旧 answer、旧 judgment 或旧 `answer-closure summary`，`重新评估当前回答`、`查看答案解析`、`回到当前回答继续修改` 也应回到当前回答而不是旧答案。
- 编辑旧 answer 的标题或正文，不应把它自动升格成当前回答；只有新建 answer 或把叶子节点切成 `answer` 时，`currentAnswerId` 才应切换。
- 如果 question 已经有一条空的当前 answer 草稿，左侧学习推进卡应切到 `当前回答草稿` 语境，主动作是 `回到当前回答继续修改`，而不是重新退回唯一 `直接回答当前问题`。
- 删除当前回答，或把当前回答切成非 `answer` 后，`currentAnswerId` 应按“优先前一个 surviving answer，再退后一个，否则清空”的顺序回退；对应 answer-closure 结果也应随失效源一起删除。
- 先生成一轮 answer-closure judgment / summary，再修改当前 answer；旧结果应保留，但 runtime 应能把它识别成“过期、建议重新评估”。同样地，先对手写 `summary` 执行 `检查这个总结`，再修改该 summary；旧 summary-check judgment 也应进入过期状态而不是被覆盖删除。
- 删除手写 `summary`，或把它切成非手写总结语义后，所有显式 `sourceSummaryId` 指向它的 summary-check judgment 都应被级联删除，不留下孤儿结果节点。

### 8. 侧栏输入止抖回归

- 修前稳定复现路径：先选中模块或深层 `question / answer`，聚焦标题输入框并连续输入超长标题；再在正文里连续输入，观察左栏“当前节点”、右栏 inspector / 学习节点引用 / 运行状态是否反复改高度并把下方卡片顶动。
- 修前重点观察项：`StructureActionBar` 的“当前节点”、`SelectedNodeInspector` 的模块 / 节点 / 路径，以及 `WorkspaceRuntimeStatusCard` 的最近结果明细是否会随标题增长、保存状态切换或提示出现 / 消失而明显回流。
- 修后确认方式：重复同一路径，确认标题、路径和保存状态仍实时同步，但这些文本只在固定槽位内更新；AI 配置卡、搜索卡等下方卡片不再被每次输入持续顶下去；输入焦点、节点选中、自动保存和运行时主路径保持可用。

### 9. question block 主视图与折叠状态

- 任选一个包含多条回答的 `question`：主视图中应按链条阅读顺序显示 `answer -> judgment -> 答案解析 -> 下一条 answer -> ...`，不要再出现“先堆所有 answer、再补 closure 结果”的割裂顺序。
- 把 `currentAnswerId` 切到另一条 answer 后，主视图顺序不应变化；只允许“当前回答”标识、runtime 主路径和对应结果归属变化，不允许整组被搬到前面。
- 手写 `summary`、`答案解析`、`总结检查结果` 必须在显示层明确区分：卡片类型标识分别显示为 `总结`、`答案解析`、`总结检查结果`，不要再出现“全部都叫总结”的混淆。
- 如果同一条 `answer` 或手写 `summary` 存在旧结果，历史必须收回各自组内：`展开 / 收起历史评估与旧解析` 只影响那条 answer，`展开 / 收起历史检查结果` 只影响那条 summary，不再存在全局“早期回答”区。
- follow-up `question` 应显示在所属这轮回答闭环之后；切换当前回答或展开旧历史时，不应被别的 answer 的旧 closure 结果挤到前面。
- 只在当前激活 block 上显示 block 级动作：`直接回答当前问题`、`插入回答`、`插入追问`、`插入总结`。切换到其他 block 或 block 内任一子节点后，动作应跟着激活 block 一起切换。
- 分别验证三种折叠：`收起 block`、节点卡片上的 `收起正文`、组内历史区的展开 / 收起。刷新后这些折叠状态应恢复，不写进 snapshot。
- 把某个 block 整体收起后，从结构树或搜索命中里点到这个 block 内的旧回答 / 历史结果 / 总结检查结果，主视图应自动展开对应 block，并在必要时一并展开正文区或所属组内历史区，让目标节点直接可见。
- 打开 runtime 页面时，左侧 `WorkspaceRuntimeActionCard` 应以“辅助概览 / 下一步建议 / 冗余入口”为主，不再抢占问答总结链主路径；与此同时，中间主视图里的 question block 动作必须保持完整可达。

## 自动化覆盖基线

- 单元 / 集成测试：Vitest
- React 组件测试：Vitest + DOM 测试环境
- 浏览器级 smoke / E2E：Playwright（Chromium）
- 类型检查：TypeScript
- 构建验证：Vite build

浏览器级 smoke 目前只做轻断言，不提前锁死仍在演进中的细节文案和复杂连续输入。

## Playwright smoke

### 安装与运行

- `pnpm install`
- `pnpm exec playwright install chromium`
- `pnpm run test:e2e`

### 当前覆盖

- 应用可正常打开，关键主区域存在，不直接落入白屏 / 不可用状态
- 首次进入无历史数据时，可初始化最小工作区并进入可编辑状态
- 对工作区做最小可见修改后，刷新仍能恢复到已保存工作区
- 预置多模块工作区后，模块切换会带来文本主视图 / 结构视图的明显反馈
- 资料区、搜索 / 定位区、导出区面板可渲染；首版只验证结构存在与最小内容展示

### 刻意未覆盖

- 真实 AI 调用与外部模型服务
- 仍在变化中的文本连续输入、标签入口、资料创建入口等交互
- 对搜索结果文案、复杂编辑行为和资源创建流程的重断言
