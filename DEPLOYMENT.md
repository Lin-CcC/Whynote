# DEPLOYMENT

本文档定义 Whynote 首版的部署基线。首版目标是个人 Web MVP，因此部署策略以静态前端优先，不引入必须存在的后端服务。

## 当前部署假设

- 运行形态：浏览器中的单页前端应用
- 构建工具：Vite
- 产物目录：`dist/`
- 数据保存：浏览器本地存储
- AI 配置：用户在应用内自行填写 OpenAI-compatible 配置

这意味着首版部署可以只发布静态文件。

## 适用范围

本文档适用于：

- 个人自用
- 小范围原型验证
- UI / 交互测试

本文档不把首版定义为公开商用方案。原因：

- 用户自填 API Key 更适合个人使用
- 没有后端代理层时，不适合作为公共 SaaS 默认形态
- 多设备同步、权限控制、平台级计费都不在首版范围内

## 构建前提

工程初始化后应满足：

- 使用 `pnpm install` 安装依赖
- 使用 `pnpm run build` 输出 `dist/`
- 使用 `pnpm run preview` 本地验证静态产物

如果引入前端路由，需要额外配置 SPA fallback，确保刷新子路径时仍回到 `index.html`。

## 配置原则

首版应尽量减少构建期强依赖环境变量。

推荐配置来源：

- 运行时用户设置：
  - `base URL`
  - `API key`
  - `model`
- 可选构建时默认值：
  - 默认 API 服务地址
  - 默认模型名
  - 默认应用标题

如需提供默认值，应只提供便捷默认，不应依赖平台私有环境变量才能启动页面。

## 本地存储限制

由于首版数据存于浏览器本地：

- 不同浏览器之间不会自动同步
- 不同设备之间不会自动同步
- 清除浏览器站点数据会丢失本地学习记录

因此建议在产品内提供显式导出路径，作为个人使用场景下的备份方式。

## 默认示例平台：Vercel

Vercel 适合当前阶段，原因：

- 对 Vite 静态站点支持直接
- Git 集成简单
- 预览部署方便 UI 与交互验证

### 推荐流程

1. 将仓库推送到 Git 平台
2. 在 Vercel 中导入仓库
3. 让平台自动识别 Vite 配置
4. 构建命令设为 `pnpm run build`
5. 输出目录设为 `dist`
6. 部署后验证首页、静态资源、前端路由与本地存储行为

### 发布后检查

- 页面首次加载正常
- 静态资源路径正确
- 本地存储可写
- 设置项刷新后保留
- 若使用前端路由，深链接刷新不报 404

## 备选平台：Cloudflare Pages

Cloudflare Pages 也是合适的静态部署目标，适合作为后续补边缘能力时的备选。

### 适合迁移到 Pages 的场景

- 后续希望接入 Cloudflare 生态
- 未来可能追加 Pages Functions / Workers 能力
- 更偏好 Cloudflare 的发布体系

### 基本流程

1. 导入仓库
2. 构建命令设为 `pnpm run build`
3. 输出目录设为 `dist`
4. 完成部署后验证 SPA 路由与存储行为

## 何时需要升级部署模型

当出现以下目标时，应重新设计部署方案，而不是继续沿用纯静态前端模式：

- 平台级 AI Key 托管
- 多用户账号与云同步
- 多设备共享工作区
- 公开商用或对外开放注册
- 需要统一审计、限流、代理或成本控制

那时应引入后端代理层，并同步更新本文件与 [DEV_NOTE.md](DEV_NOTE.md)。
