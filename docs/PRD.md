# Whynote PRD

## 1. 产品概述

Whynote 是一个围绕提问式学习构建的节点化学习系统。它把学习过程中的问题、回答、总结、判断、资料引用和结构调整统一到同一棵节点树中，让用户在学习的同时自然生成可整理、可导出的笔记。

Whynote 首版不是通用思维导图工具，也不是普通聊天式 AI 笔记软件。它的核心目标是把“思考过程”变成“可持续编辑的结构化文档”。

## 2. 目标与边界

### 2.1 首版目标

- 验证“边学习边长笔记”的核心体验
- 验证统一节点模型是否足以承载学习计划、问答、总结、判断和资料引用
- 验证文本主视图与结构视图联动是否比传统聊天工具更适合长期学习
- 验证自动结构化结果是否能被用户轻量修正，而不是打断学习节奏

### 2.2 首版非目标

- 不做通用自由画布思维导图
- 不做公开商用级账号与平台代理层
- 不做云同步与多人协作
- 不做自动联网资料检索
- 不做 PDF / DOC 导出

## 3. 目标用户

首版优先服务个人自用场景，尤其是：

- 习惯通过提问来推进理解的人
- 需要在学习过程中不断重排笔记结构的人
- 既想要 AI 引导，又不想失去对自己笔记结构掌控权的人

## 4. 产品原则

- 节点统一：所有内容都以节点组织
- 类型清晰：节点统一，但语义必须明确
- 文本优先：文本区是主舞台，结构视图是辅助与重排工具
- 自动优先但可回退：系统先自动组织，用户再轻量修正
- 少 UI：只保留对学习链路真正必要的功能
- 过程与成稿分层：学习过程可见，但导出结果默认更像整理后的笔记

## 5. 核心产品模型

### 5.1 节点层级

Whynote 首版使用如下固定树结构：

```text
theme-root
├─ module
│  ├─ plan-step
│  │  ├─ question
│  │  │  ├─ question
│  │  │  ├─ answer
│  │  │  ├─ summary
│  │  │  └─ judgment
│  │  ├─ answer
│  │  ├─ summary
│  │  └─ judgment
│  └─ summary
└─ resource
   └─ resource-fragment
```

### 5.2 节点类型定义

- `theme-root`
  整个学习主题。根节点下直接子级只允许 `module` 和 `resource`。
- `module`
  一个学习方向模块。它既是结构单元，也是可单独导出的模块文档。
- `plan-step`
  学习计划步骤。模块内学习内容的主要容器。
- `question`
  问题节点。可作为主问题，也可作为追问或自动拆分出的子问题。
- `answer`
  回答节点。允许同一问题下存在多个回答节点。
- `summary`
  总结节点。允许阶段性总结与模块级总结共存。
- `judgment`
  判断 / 评估节点。用于判断用户回答或总结的准确性与完整性。
- `resource`
  资料源节点。一个 `resource` 对应一份被学习系统持续引用的资料来源，可以来自 URL、本地上传或手动录入。它承接用户可理解的标题、资料概况与必要来源信息，并作为后续正文解析、摘要和引用的统一宿主。
- `resource-fragment`
  资料切片 / 摘录节点。它不是任意临时笔记片段，而是挂在某个 `resource` 下、可被重复引用与稳定回跳的资料锚点。只有当系统或用户需要对某段内容做稳定引用时才生成，不要求每份资料都提前切满全文。

### 5.3 节点标签

首版所有节点都允许挂标签。内建标签至少包括：

- `重要`
- `未理解`
- `待验证`
- `待整理`

模型层不限制标签只能作用于某类节点；显示层可对不同节点类型做轻重区分。

## 6. 模块与学习计划生成

### 6.1 入口

首版支持三种学习入口：

- 只输入主题
- 导入资料（URL / 本地 `txt / md` / 手动资料卡）
- 导入资料并附带用户问题 / 学习偏好

第三种是推荐路径，因为它最容易生成针对性的学习结构。

### 6.2 学习模式

首版支持三种模式：

- `快速`
  - 2-3 个 `module`
  - 每个模块 2-3 个 `plan-step`
- `标准`
  - 3-5 个 `module`
  - 每个模块 3-5 个 `plan-step`
- `深度`
  - 4-7 个 `module`
  - 每个模块 4-6 个 `plan-step`

默认模式为 `标准`。

### 6.3 计划节点规则

- 学习计划进入节点树，`plan-step` 是正式节点
- `plan-step` 编辑时可见
- 导出默认隐藏 `plan-step` 标题，但保留其子内容顺序并并入模块正文
- 用户可显式选择包含完整学习路径

### 6.4 步骤完成状态

- `plan-step` 至少支持 `todo / doing / done`
- 完成状态以用户手动确认为准
- 系统可根据当前步骤下的问题、总结与判断情况给出“建议完成”提示
- 系统不自动将步骤置为完成

## 7. 问题与自动结构化

### 7.1 复合问题自动拆分

用户输入包含多个独立问题的复合问题时：

- 系统默认自动拆分
- 原句保留为父 `question`
- 子问题显式生成并可直接编辑
- 用户可拖拽、升降级或撤销拆分结果

### 7.2 子问题排序

- 优先按理解依赖排序
- 若系统对依赖关系低置信度，则退回原句顺序
- 用户手动排序优先于系统结果

### 7.3 追问规则

- 用户在学习中提出追问时，默认挂到当前 `plan-step` 或当前 `question` 下
- 子问题完成后，父问题仍保留，可继续回答、总结或判断

### 7.4 回答 / 总结 / 判断节点

- `answer / summary / judgment` 都是独立节点，不是问题内部的文本区块
- 同一问题下允许存在多个同类型节点，以保留迭代学习过程

## 8. 结构编辑

### 8.1 通用操作

首版树编辑使用统一的结构操作：

- `提升一级`
- `降低一级`
- 拖拽重排

### 8.2 根层模块语义

- 节点提升到根节点直接子级时，自动获得 `module` 语义
- 标题默认沿用当前节点标题
- 原有子树、标签、引用与顺序保留
- `resource` 系节点不参与此转换

### 8.3 产品意图

结构编辑是 Whynote 的核心能力之一，用于让用户以最少文本搬运成本重新组织自己的学习材料。

## 9. 资料区与引用

### 9.1 全局资料区

- 所有资料源统一放在根节点下的全局资料区
- 资料不直接复制到模块内部
- 模块、步骤、问题、回答、总结、判断通过引用连接资料

### 9.2 `resource` / `resource-fragment` 的正式心智

- `resource` 是一份资料的长期宿主，而不是临时输入框。一个 `resource` 对应一条稳定资料来源。
- `resource` 负责承接标题、资料概况、必要来源信息，以及后续可供引用的正文基础。
- `resource-fragment` 是某个 `resource` 内可被反复引用的稳定锚点，不是随手记下的一段零散笔记。
- `resource-fragment` 至少应能说明“摘了什么、来自哪份资料、回跳到哪里”；它可以是原文摘录，也可以是对某段内容的压缩摘记。
- 不是每份资料都必须先拥有多个 `resource-fragment`。只有当系统或用户需要稳定引用某段内容时才生成。

### 9.3 资料导入流程

- 首版保留三类资料入口：URL 录入、本地 `txt / md` 上传、手动创建 `resource`。
- 系统应先创建一个可见的 `resource` 壳节点，再异步补齐标题、概况、来源信息与可用的正文基础，避免导入失败时资料完全消失。
- URL 导入时，系统优先在能抽取到网页内容的前提下补齐页面标题、页面概况与必要来源信息；若抽取失败，保留 URL 作为来源并允许用户补录。
- 本地 `txt / md` 上传时，系统优先根据文件名、首级标题与正文内容补齐标题、概况与后续引用所需的正文基础。
- 用户手动修改过的标题 / 概况优先级高于自动结果，后续重试导入或重新摘要不能静默覆盖用户改写。

### 9.4 自动摘要流程

- 自动摘要的输出至少包括：资料标题、资料概况、必要来源信息。
- URL 资料的必要来源信息至少应包含可见 URL、站点 / 域名与导入时间；后续可补 canonical URL、作者、发布时间。
- 本地资料的必要来源信息至少应包含文件名、导入时间与 MIME / 类型提示。
- 资料概况应优先回答“这份资料讲什么、适合在什么问题下被引用”，而不是生成泛泛摘要。
- 自动摘要是默认起点，不是不可编辑结果。用户可在 `resource` 上直接修订。

### 9.5 自动摘录流程

- 长期主流程不是“导入资料后立刻手工新建摘录”，而是“先有 `resource`，再在需要稳定引用时生成 `resource-fragment`”。
- `resource-fragment` 的系统自动生成主要发生在三类场景：
  - 导入后生成少量高置信度候选片段（第二层能力）
  - 学习节点需要精确引用某段资料时
  - 搜索 / 对照阅读命中后，需要固化可复用锚点时
- 首版 MVP 不做导入即全文切片，也不要求每段正文都预生成 fragment。
- 手动创建 `resource-fragment` 仍然保留，但主要用于自动处理没覆盖到的补摘、纠偏和精修。

### 9.6 学习过程中的自动引用

- `question / answer / summary / judgment` 在引用资料时，应优先关联已有 `resource-fragment`，而不是重复造一个新摘录。
- 如果已有 fragment 无法覆盖当前引用，但系统能在某个 `resource` 的正文基础中定位到清晰片段，则允许自动创建新的 `resource-fragment` 并立刻建立引用关系。
- 如果系统只能判断“这段学习内容与某份资料相关”，却无法给出可靠片段定位，则应退回到 `resource` 级引用，而不是伪造精确 fragment。
- fragment 去重至少基于同一 `resource` 下的定位信息、正文相似度与摘要内容复用，避免同一句话被反复切成多个节点。
- 首版稳定回跳点先以 `resource` / `resource-fragment` 节点为准；后续阅读器再把 fragment 深化为资料正文内部的精确定位。

### 9.7 手动摘录入口的定位

- 当前“新建摘录”与“新建资料”并列，属于过渡态入口。
- 最终产品里，手动摘录应降级为 `resource` 详情内的补充动作，或引用纠偏流程中的高级操作，不应继续作为资料区的主入口。
- 真正的主流程应是：导入 `resource` -> 系统补齐摘要 / 来源 -> 学习时自动引用 / 自动生成 fragment -> 用户只在需要时补摘或修正。
- 手动摘录仍有保留价值：不支持自动解析的资料、需要人工裁剪的片段，以及系统误判后的修正。

## 10. 搜索、定位与标签

### 10.1 搜索作用域

- 默认搜索范围是当前模块
- 搜不到时提示“当前模块无结果”
- 系统不自动扩大到全主题
- 用户可主动切换到“全主题”或“资料区”搜索

### 10.2 定位表现

- 当有多个命中结果时，使用类似编辑器定位栏的可视化相对位置提示
- 点击定位栏中的命中位置，可跳转到对应节点

### 10.3 标签检索

- 标签可作为搜索与筛选维度
- 导出按标签筛选时，应保留命中节点和必要祖先路径，避免上下文丢失

## 11. 导出规则

首版支持导出：

- 整个主题
- 单个模块
- 按标签筛选后的结果

格式仅限：

- Markdown
- TXT

默认导出行为：

- 隐藏 `plan-step` 标题
- 保留 `plan-step` 下的子内容顺序
- 生成更接近整理后笔记的文本

用户可显式选择导出完整学习路径。

## 12. 界面基线

### 12.1 文本主视图

- 文本区是学习主舞台
- 用户的大部分输入、修订、总结、判断发生在文本区

### 12.2 结构视图

- 用于展示节点结构
- 用于拖拽重排、升降级、切换关注范围
- 不是自由画布式思维导图

### 12.3 全局资料区

- 独立于模块结构存在
- 作为引用源与对照学习的资料容器

### 12.4 搜索与定位栏

- 搜索默认作用于当前模块
- 多命中结果通过定位栏显示相对位置与跳转入口

## 13. AI 设置基线

首版采用用户自填 OpenAI-compatible 配置，至少支持：

- `base URL`
- `API key`
- `model`

首版定位是个人模式，不把该方案定义为公开商用默认形态。

## 14. 存储基线

- `IndexedDB`
  - 工作区
  - 模块
  - 节点树
  - 标签
  - 引用关系
  - 资料元数据
  - 资料正文基础
- `localStorage`
  - 设置项
  - 最近打开状态
  - 界面偏好

## 15. 必须覆盖的场景

- 用户输入主题后，系统生成模块与计划步骤
- 用户导入资料并提出问题后，系统生成更有针对性的学习结构
- 用户录入 URL 后，系统自动补标题、概况与必要来源信息；若抽取失败，仍保留 `resource` 并允许用户补录
- 用户上传 `txt / md` 后，系统自动补标题、概况与可供后续引用的正文基础
- 用户输入复合问题后，系统自动拆分并继续逐个引导
- 用户通过提升 / 降低和拖拽重排结构
- 节点提升到根层后自动转为模块
- 用户在 `question / answer / summary / judgment` 中引用资料时，系统优先复用已有 fragment，不足时按需生成新 fragment
- 用户在当前模块内搜索、切全主题搜索、切资料区搜索
- 用户导出模块时默认隐藏步骤标题，但保留学习内容

## 16. 资料能力分层演进

### 16.1 第一层：首版 MVP

- URL 手动录入
- URL 自动补标题 / 概况 / 基础来源信息
- 本地 `txt / md` 上传
- 手动创建 `resource`
- 手动创建 `resource-fragment` 保留，但降级为补充入口

### 16.2 第二层：自动化增强

- 导入后自动生成 fragment 候选
- 学习过程中自动创建 / 复用 fragment
- 更强的资料摘要与来源补全

### 16.3 第三层：后续能力

- `PDF / DOCX`
- 完整资料阅读器
- 引用对照区
- 外部资料检索
