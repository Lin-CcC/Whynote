# V1.3 结构地图演示收尾

这份文档只记录一条可直接讲述、可直接复现的结构地图演示路径，不再重复组件级细节。

## 选用的真实 module

- 主题：`结构地图演示主题`
- 模块：`问题块模块`
- 运行方式：真实 `WorkspaceRuntimeScreen` + IndexedDB 持久化恢复，不走孤立组件 demo

这个模块刻意同时包含：

- `question block`
- 两轮 `answer` 闭环
- `follow-up question`
- 手写 `summary`
- `summary-check judgment`
- `stale` 结果与显式 `source pairing`

这样一条路径就能把结构地图最容易失真的几种语义一起验掉。

## Walkthrough

1. 打开运行时工作区，进入 `问题块模块`。
2. 在文档视图确认主问题已经存在：
   - 当前回答是 `第二版回答`
   - 有一条 `追问：下一轮问题`
   - 有一组手写 `我的总结`
   - `第二版最新解析` 与 `最新总结检查结果` 处于 `已过期`
3. 切到 `结构地图`。
4. 拖动 `第一版回答` 的 answer group 到 `question-main` 的尾部落点。
5. 拖动 `追问：下一轮问题` 到 `question-main` 的头部落点。
6. 拖动 `我的总结` 的 summary group 到 `question-main` 的头部落点。
7. 每次 drop 后都自动回到文档视图，继续看真实文档顺序与语义是否一致。

本次实际观察到的最终文档顺序为：

1. `我的总结`
2. `最新总结检查结果`
3. `追问：下一轮问题`
4. `追问回答`
5. `第二版回答`
6. `第二版最新评估`
7. `第二版最新解析`
8. `第一版回答`
9. `第一版最新评估`
10. `第一版最新解析`

对应持久化后的 `question-main.childIds` 也同步写回为：

1. `summary-manual`
2. `judgment-summary-old`
3. `judgment-summary-latest`
4. `question-follow-up`
5. `answer-second`
6. `judgment-second-latest`
7. `summary-second-latest`
8. `answer-first`
9. `judgment-first-history`
10. `summary-first-history`
11. `judgment-first-latest`
12. `summary-first-latest`

## 手工回归清单

- 从文档切到结构地图，再从结构地图回到文档，主视图切换正常。
- 三次拖动都出现合法落点提示，drop 后自动回到文档。
- 文档顺序与持久化 childIds 同步变化，没有出现“地图顺序变了、正文没变”的分叉。
- `question block` 仍可正常收起 / 展开，没有因为地图写回破坏折叠行为。
- `currentAnswerId` 仍稳定指向 `第二版回答`，`第一版回答` 没有被误升格。
- `第二版最新解析` 仍显示 `配对回答：当前回答 · 第二版回答`。
- `最新总结检查结果` 仍显示 `检查对象：我的总结`。
- `第二版最新解析` 与 `最新总结检查结果` 的 `已过期` 标记仍在，没有因为重排被错误清掉。
- follow-up 仍然是独立 question block，不会并进 answer group。

## 结构地图解决的实际编排问题

- 可以把手写总结整组提前，先交代“我目前怎么总结这段内容”，再展开后续问答链。
- 可以把 follow-up 前置，让“下一轮真正要追的问题”比旧回答链更早出现。
- 可以把旧 answer 闭环整体后移，避免旧回答长期占据主阅读入口。
- 整个过程只改树顺序，不改 `currentAnswerId`、source pairing 或 stale 语义，说明结构地图确实是“编排层”而不是“语义重写层”。

## 当前已知边界

- 仓库默认 demo workspace 不足以同时覆盖 `source pairing`、`stale` 与 follow-up 闭环，因此这次演示使用运行时 seed 的真实工作区快照，而不是内置 demo 模块。
- 这次收尾不新增录屏、截图或素材目录；演示资产仍以仓库文档 + 可复现路径为主。
- 这份 walkthrough 只覆盖单个模块内的 question block 重排，不扩成跨模块叙事重组。
