# V1.1 第一轮：问题链条编辑器

## 目标

V1.1 第一轮只收 `question / answer / 手写 summary` 这条主线，把当前系统从“可运行学习壳层”推进成一个：

- 主视图为主
- 问题链条为核心
- AI 作为编辑辅助

的学习编辑器。

本轮明确不做：

- 思维导图 / 拖拽编排 / 图形画布视图
- 主视图与导图双表面同步
- `judgment / hint / 查看答案解析` 的主视图重做
- 完整只读 / 编辑双模式
- 快捷键体系 / 命令面板
- 资料区深嵌主视图
- AI 结构整理、重排、压缩等强协同编辑动作

## 核心决策

- 主视图是高频操作主入口；左侧 runtime action card 降成辅助概览
- 多条 `answer` 属于正文内容；`question.currentAnswerId` 显式指向当前回答
- `summary` 显示层明确分三类：
  - `答案解析`
  - `总结`
  - `总结检查结果`
- `judgment / 答案解析 / 总结检查结果` 与其服务对象采用显式 `sourceId + sourceUpdatedAt` 配对
- 折叠状态属于本地视图状态，跨刷新保留，但不写进 snapshot
- 导出默认仍是完整导出；“仅当前展开内容”第一轮只服务主编辑链条导出

## 数据契约

### QuestionNode

- 新增：`currentAnswerId?: string`

语义：

- 表示当前问题当前采用哪一版 `answer` 作为主路径工作对象
- `直接回答当前问题` 生成新 answer 时自动写入
- 手动 `插入回答` 生成新 answer 时自动写入
- 叶子学习节点切换成 `answer` 时自动写入
- 编辑旧 answer 不会自动升格成当前回答

兼容规则：

- 旧 question 缺少 `currentAnswerId` 时，首次进入 V1.1 默认取“最新有内容 answer”
- 若没有任何有内容 answer，则保持为空

失效回退：

1. 若当前回答被删除、切成非 `answer` 类型、或不再属于该 question
2. 优先回退到树顺序上更靠前的最近一条 surviving answer
3. 没有则回退到更靠后的第一条 surviving answer
4. 仍没有则清空 `currentAnswerId`

### SummaryNode

- 新增：
  - `sourceAnswerId?: string`
  - `sourceAnswerUpdatedAt?: string`

规则：

- 仅对 `answer-closure` summary 使用
- 手写 `manual` summary 不要求这两个字段

### JudgmentNode

- 新增：
  - `sourceAnswerId?: string`
  - `sourceAnswerUpdatedAt?: string`
  - `sourceSummaryId?: string`
  - `sourceSummaryUpdatedAt?: string`

规则：

- `answer-closure judgment` 必须写 `sourceAnswerId + sourceAnswerUpdatedAt`
- `summary-check judgment` 必须写 `sourceSummaryId + sourceSummaryUpdatedAt`
- 若某次 summary-check 有明确 answer 上下文，建议一并写 `sourceAnswerId + sourceAnswerUpdatedAt`

## 行为规格

### 1. question block

`question block` 是**显示 / 交互组织**，不是新节点类型。

- follow-up question 仍是真实子节点
- 底层树模型不变
- 主视图允许为 question block 做显示层重组

一个 question block 以当前 question 为锚点，视觉上连续包含：

- question
- 当前回答区
- 与当前回答对应的 `judgment / hint / 答案解析`
- 手写 `总结`
- 连续展示的 follow-up 下游子链

当前激活 block 的判定规则：

- 由当前选中节点向上追溯到最近的 question 决定
- 选中 `question / answer / judgment / 答案解析 / 手写总结 / 总结检查结果` 中任一节点，都应激活同一个 block

### 2. 多回答与当前回答

同一 `question` 下允许存在多条 `answer`，它们共同构成学习 / 修订过程，属于正文内容。

runtime 默认围绕 `question.currentAnswerId` 工作，而不是围绕：

- 最新 answer
- 当前选中的 answer

切换当前回答的唯一入口：

- 用户显式执行 `设为当前回答`
- `直接回答当前问题` 生成新 answer
- 手动 `插入回答` 生成新 answer
- 叶子学习节点通过类型切换变成 `answer`

额外规则：

- 编辑旧 answer 不会自动升格成当前回答
- `currentAnswerId` 指向的 answer 即使正文为空，也视为“已有当前草稿”
- 这时 question block 主动作应优先引导继续写 / 编辑，而不是继续把 `直接回答当前问题` 当主动作

主视图呈现：

- 当前回答作为主内容区
- 其余 answer 进入 `早期回答 / 旧版本` 折叠区，默认收起
- 旧回答区按树里的真实顺序展示

### 3. 显式配对与过期语义

第一轮不再把“哪条 judgment / 解析属于哪条 answer / summary”主要交给 sibling 顺序去猜。

新生成节点必须尽量显式写入：

- `answer-closure judgment.sourceAnswerId = 当前回答 id`
- `answer-closure judgment.sourceAnswerUpdatedAt = 当前回答 updatedAt`
- `answer-closure summary.sourceAnswerId = 当前回答 id`
- `answer-closure summary.sourceAnswerUpdatedAt = 当前回答 updatedAt`
- `summary-check judgment.sourceSummaryId = 被检查 summary id`
- `summary-check judgment.sourceSummaryUpdatedAt = 被检查 summary updatedAt`
- 若该次 summary-check 有明确 answer 上下文，则一并写 `sourceAnswerId + sourceAnswerUpdatedAt`

主视图与 runtime 组装规则：

- question block 的判断区 / 解析区只展示与 `currentAnswerId` 匹配的最新结果
- 旧回答对应的 judgment / 答案解析进入历史区，默认收起
- 手写 summary 的当前检查结果取 `sourceSummaryId === 当前 summary id` 的最新一条

过期判定：

- 若当前源节点 `updatedAt` 晚于结果节点记录的 `source...UpdatedAt`
- 则该 judgment / 答案解析 / 总结检查结果标记为过期

过期处理：

- 不自动删除
- 继续保留在当前区
- 但要明确显示“结果已过期，建议重新评估 / 重新检查”

### 4. 历史保留与级联清理

同一条 answer 多次重评：

- 保留历史 judgment / 答案解析
- 主视图只突出最新一轮
- 旧轮次进入历史区

同一条 summary 多次检查：

- 保留历史检查结果
- 主视图只突出最新一条
- 更早结果进入历史区

但源节点失效时，优先执行级联清理：

- 当 answer 被删除，或被切换成非 `answer` 类型时：
  - 所有 `sourceAnswerId` 指向它的 answer-closure judgment / summary 一起级联删除
- 当手写 summary 被删除，或被切换成非手写总结语义时：
  - 所有 `sourceSummaryId` 指向它的 summary-check judgment 一起级联删除

不保留孤儿结果节点。

### 5. 主视图动作

`question` 相关动作不继续硬塞进现有单节点 inline 机制；V1.1 第一轮新增一个 **block 级 action surface**。

#### question block 级动作

- `直接回答当前问题`
- `插入回答`
- `插入追问`
- `插入总结`

#### answer 节点级动作

- `重新评估当前回答`
- `继续修改`
- `设为当前回答`（仅对非当前回答显示）

#### 手写 summary 节点级动作

- `检查这个总结`
- `继续修改`

第一轮不重做：

- `judgment`
- `hint`
- `查看答案解析`

它们继续沿用现有 inline / runtime 入口。

### 6. summary 三种显示语义

不新增节点类型，但显示层必须明确区分：

- 系统生成、服务当前回答解析的 `summary`：`答案解析`
- 用户手写、服务链条收束的 `summary`：`总结`
- `summary-evaluation` 产出的 `judgment`：`总结检查结果`

同一 question block 内允许同时存在：

- `答案解析`
- `总结`
- `总结检查结果`

它们互不替代。

### 7. 折叠

第一轮支持的折叠粒度：

- 整个 `question block`
- `answer / judgment / summary` 的正文内容区
- `早期回答 / 历史评估 / 历史检查结果` 历史区

折叠状态属于本地视图状态：

- 跨刷新恢复
- 按 workspace 本地保存
- 建议存入 `UiPreferences.values.workspaceViews[workspaceId]`
- 不进入 snapshot，不算正文语义

折叠状态与左侧 `StructureTree.expandedNodeIds` 彼此独立。

如果用户从结构树 / 搜索 / 运行时入口选中了折叠 block 内的节点：

- 主视图自动展开对应 block，使目标节点可见

### 8. 导出

导出默认仍是 `全部内容`，忽略折叠状态。

新增显式选项：`仅当前展开内容`

第一轮适用范围：

- 仅对 `current-module / theme` 生效
- `filtered` 导出继续维持完整导出语义，不接入折叠状态

“仅当前展开内容”规则固定为：

- 整个折叠的 `question block`：整块不导
- 折叠的节点正文：导出节点标题 / 类型，但不导正文
- 折叠的 `早期回答 / 历史评估 / 历史检查结果`：整段历史区不导
- 被省略正文的节点，其附属引用内容也不导
- 第一轮不额外裁剪 root 级资源树；折叠导出只作用于学习链条视图

## 测试与验收

至少需要覆盖：

### 当前回答语义

- 多 answer 场景下，`question.currentAnswerId` 控制 runtime 主路径
- 选中旧 answer 不自动篡改当前回答
- `设为当前回答` 后，评估 / 解析 / 回改都切到该 answer
- 手动插入 answer 和类型切成 answer 都自动晋升为当前回答
- 空当前 answer 草稿会把主路径切成继续写 / 编辑，而不是 direct answer
- 删除 / 改类型导致当前回答失效时，fallback 规则正确
- 旧工作区首次加载时，默认取“最新有内容 answer”为当前回答

### 显式配对语义

- 新生成的 answer-closure judgment / summary 写入 `sourceAnswerId + sourceAnswerUpdatedAt`
- 新生成的 summary-check judgment 写入 `sourceSummaryId + sourceSummaryUpdatedAt`
- question block 只展示与 `currentAnswerId` 匹配的最新 judgment / 答案解析
- 同一 summary 只突出显示最新一条 summary-check 结果
- 缺失 `source*` 的旧数据仍能通过 legacy heuristic 正常显示

### 过期与级联清理

- 修改当前 answer 后，旧 judgment / 答案解析留在当前区并标记过期
- 修改手写 summary 后，旧总结检查结果留在当前区并标记过期
- 删除 answer / 改成非 answer 后，其 answer-closure judgment / summary 被级联删除
- 删除手写 summary / 改成非手写总结语义后，其 summary-check judgment 被级联删除

### 主视图与折叠

- `question block` 可直接回答、插入回答、插入追问、插入总结
- `answer` 可重新评估、设为当前回答、继续修改
- 手写 `summary` 可检查理解并继续修改
- question block 折叠 / 展开稳定
- answer / judgment / summary 正文可折叠
- 旧回答区、历史评估区、历史检查区默认收起且可展开
- 折叠状态跨刷新本地恢复
- 选中折叠块内节点时自动展开 block
- 主视图折叠与左侧结构树展开状态互不干扰

### 导出

- 默认导出完整内容
- 选择“仅当前展开内容”后，折叠块 / 折叠正文 / 折叠历史区按规则省略
- `filtered` 导出不受折叠状态影响
