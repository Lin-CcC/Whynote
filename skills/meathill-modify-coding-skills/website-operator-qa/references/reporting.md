# 报告与问题记录格式

## 建议输出结构

### 1. 测试目标与范围

- 测试角色：运营人员、客户、委托方、内容编辑
- 测试站点：前台 URL、CMS URL
- 测试边界：是否包含后台、是否包含多语言、是否包含移动端、是否允许创建数据

### 2. 测试用例

- 用业务语言描述用例，不要写成自动化断言
- 每条用例最好包含：场景、目标、入口、预期业务结果
- 可按首页、内容发现、转化、CMS、预览、多语言、移动端分组

### 3. 执行结果

- 每条用例都标记为 `通过`、`部分通过`、`失败`
- `部分通过` 必须说明哪些部分可用、哪些部分不可用
- 优先写出运营或客户真正看到的结果，再补充技术证据

### 4. 问题记录

每个问题至少包含以下信息：

- 标题：一句话说清用户可见问题
- 严重性：高 / 中 / 低
- 页面或模块：URL、CMS 集合或具体功能入口
- 重现步骤：尽量短、可重复
- 实际结果：用户或运营看到什么
- 预期结果：本应如何工作
- 业务影响：为什么这会影响对外服务、留资、发布或维护效率
- 辅助证据：必要时补充控制台、网络请求、代码路径

## GitHub Issue 书写原则

- 一条 issue 只写一个问题，不要把多个症状堆在一起
- 标题写用户可见问题，不要只写底层原因
- 正文优先写：环境、步骤、实际结果、预期结果、业务影响
- 只有在确认有帮助时再补充代码线索、日志或可疑根因
- 如果是脏数据问题，也要写清楚为什么系统允许它流入线上

## 严重性判断

- 高：直接阻断留资、购买、发布、预览，或让对外页面明显失效
- 中：明显损伤可用性、可信度或运营效率，但存在绕行方案
- 低：不阻断主流程的细节问题、资源缺失或控制台噪音

## 交付时别遗漏

- 明确哪些测试数据已清理，哪些未清理
- 明确哪些问题已建 issue，哪些还只是本轮记录
- 明确哪些结论来自浏览器观察，哪些来自代码或网络日志辅助确认
